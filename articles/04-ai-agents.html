AIエージェント設計パターン

LLMに「考えて行動させる」システム設計

14 min read 2026-03-25 エージェント Tool Use ReAct マルチエージェント

「ChatGPTに指示すれば何でもやってくれる」——そう期待したものの、1回の質問・回答では完結しない複雑なタスクに対して限界を感じた開発者は多いはずです。コードを書かせて実行させてエラーを直させる、ドキュメントを検索してまとめてメールの下書きを作る——こうした複数ステップの自律的な作業を可能にするのがAIエージェントです。本記事では、AIエージェントの設計パターンを実装観点から体系的に解説します。

1. エージェントとは何か

AIエージェントを正しく設計するには、まず「チャットbot」との本質的な違いを理解する必要があります。

チャットbotは「入力に対して1回応答を返す」静的な処理です。一方、AIエージェントは思考→ツール実行→結果の観察→再思考というループを自律的に繰り返し、目標を達成するまで処理を継続します。外部APIの呼び出し、コードの実行、ファイルの操作といった「行動」を伴う点が根本的に異なります。

エージェントが持つ3つの能力

  • 計画（Planning） — タスクを複数のサブタスクに分解し、実行順序を決定する能力。「Pythonスクリプトを書いて実行し、エラーがあれば修正する」という計画を立てられる
  • ツール使用（Tool Use） — 外部システム（API、データベース、コード実行環境）を呼び出して情報を取得・操作する能力。LLM単体では持てない「行動力」を与える
  • メモリ（Memory） — 短期メモリ（会話コンテキスト）と長期メモリ（ベクトルDB・KV store）を使い分けて、過去の作業状態を参照する能力

実用例：コード生成の自律ループ

AIエージェントの分かりやすい例が、コード生成・実行・デバッグの自律ループです。開発者が「CSVを読み込んで売上グラフを生成するPythonスクリプトを書いて」と指示すると、エージェントは①コードを生成し②サンドボックスで実行し③エラーメッセージを観察して④修正を加え⑤成功するまでループします。人間の介入なしに、数回の試行で動作するコードを納品できます。

2. ReActパターンとTool Use

AIエージェントの最も基礎的かつ普及している設計パターンがReAct（Reason + Act）です。2022年にPrinceton大学とGoogle Researchの共同研究で発表された手法で、LLMに「思考の言語化」と「行動」を交互に行わせることで、複雑なタスクに対する性能を大幅に向上させました。

Function Calling / Tool Use の定義

現代のLLM APIはFunction Calling（またはTool Use）という仕組みを提供しており、開発者が用意したツールの定義をLLMに渡すと、LLMが「このツールをこのパラメータで呼び出す」という判断を下せるようになります。

json — Tool定義のスキーマ例 
{
  "name": "search_documents",
  "description": "社内ドキュメントを検索する",
  "parameters": {
    "type": "object",
    "properties": {
      "query": {
        "type": "string",
        "description": "検索クエリ（自然言語で記述）"
      },
      "limit": {
        "type": "integer",
        "description": "返す結果の最大件数",
        "default": 5
      }
    },
    "required": ["query"]
  }
}

Pythonによるエージェントループの実装

実装は chat.completions.create(tools=TOOLS, tool_choice="auto") を呼び出し、message.tool_calls が返ってきた場合はツールを実行して結果をメッセージ履歴に追加、返ってこなければ最終回答として返すループを組むだけです。無限ループ防止に max_iterations の上限設定は必須です。

3. マルチエージェントアーキテクチャ

単一のエージェントでは処理しきれない複雑なタスクや、並列実行でスループットを上げたいシナリオでは、マルチエージェントアーキテクチャが有効です。代表的な3つのパターンを紹介します。

3つのアーキテクチャパターン

1. Single Agent — 1つのエージェントがすべてのツールを持ち、すべてのタスクを処理します。実装がシンプルで、小〜中規模のタスクに適しています。ツールが増えすぎるとコンテキストが肥大化し、性能が下がる欠点があります。

2. Orchestrator + Workers — オーケストレーターエージェントがタスクを分解し、専門化されたワーカーエージェントに割り振ります。ワーカーは並列実行でき、役割分担によって各エージェントが持つコンテキストを小さく保てます。中〜大規模の業務自動化に適しています。

3. Hierarchical — オーケストレーターが複数のサブオーケストレーターを持ち、それぞれが複数のワーカーを管理する多階層構造です。非常に複雑なタスク分解が可能ですが、設計・デバッグコストが高くなります。

パターン比較

パターン 複雑さ 適用シナリオ コスト傾向
Single Agent 低 PoC、単純タスク自動化 低
Orchestrator + Workers 中 並列処理が必要な中〜大規模業務 中〜高
Hierarchical 高 大規模・複雑なタスク分解が必要な場合 高

4. ツール設計の原則

エージェントに与えるツールの設計品質は、システム全体の信頼性に直結します。「動くツール」ではなく「安全で信頼性の高いツール」を設計するための原則を解説します。

べき等性（Idempotency）

べき等性とは「同じ操作を何度実行しても結果が同じになる」性質です。エージェントはネットワークエラーやLLMの判断ブレなどで同じツールを複数回呼び出すことがあります。データ挿入ツールが冪等でないと、同じレコードが重複して登録される事故につながります。

ツールの粒度

ツールが細かすぎると管理対象が増えLLMがどれを使うか迷います。粗すぎると柔軟性が下がり、LLMが意図しない大きな操作を一括実行するリスクが生まれます。「1ツール1責務」を原則とし、操作の結果が観測・確認できる粒度に収めることが推奨です。

エージェントに与えてはいけないツール

以下の不可逆な操作をツールとして直接エージェントに与えることは避けてください。誤った判断やループ暴走時の被害が取り返しのつかないものになります。

  • 本番データベースへの直接DELETE / DROPクエリ
  • メール・通知の自動送信（大量送信事故のリスク）
  • クラウドリソースの削除・変更（S3バケット削除、EC2終了など）
  • 外部決済・契約APIの実行
  • 本番環境へのデプロイコマンド

これらが必要な場合は必ずHuman-in-the-loopで人間の承認を挟むか、ステージング環境のみを対象としたツールを提供してください。

5. リスクと制御

エージェントは自律的に動作するため、通常のAPIやバッチ処理とは異なる種類のリスクを持ちます。設計段階でこれらのリスクを認識し、制御機構を組み込むことが安全な運用の前提です。

主要リスクと対策

リスク 対策 実装方法
ループ暴走 最大反復数制限 max_iterations=10
コスト爆発 トークン予算管理 token_budget=50000
不正ツール実行 承認ゲート（Human-in-the-loop） human_approval=True
機密情報漏洩 出力フィルタリング PII検出 + マスク処理

ループ暴走とコスト爆発への対策

エージェントが特定の状況でループから抜け出せなくなる「ループ暴走」は最も頻発する問題です。原因はツールが常に失敗を返す、LLMが同じアクションを繰り返す判断をするなど様々です。最大反復数の上限設定は必須です。またトークン使用量は非同期でモニタリングし、予算超過時に即座に停止する仕組みも併せて実装します。

Human-in-the-loop設計パターン

高リスク操作（メール送信、本番DB更新、外部APIコール等）の前に人間の確認を挟むHuman-in-the-loopパターンは、エージェントを業務で安全に活用するための重要な設計です。エージェントは「提案」を生成し、人間が「承認・却下・修正」を行う、という役割分担にすることで、意思決定の責任の所在を明確にできます。

この記事のまとめ

  1. AIエージェントはチャットbotと異なり、思考→ツール実行→観察のループを自律的に繰り返します。計画・ツール使用・メモリの3能力を持ち、複数ステップのタスクを自律実行できます
  2. ReActパターンはエージェント設計の基本。Reason（思考）→ Act（行動）→ Observe（観察）のループをFunction Callingで実装します。LLMにツール定義スキーマを渡すだけで判断させられます
  3. 複雑なタスクにはOrchestrator + Workersパターンが有効。専門化されたワーカーに役割を分担させることで、各エージェントのコンテキストを小さく保ちつつ並列処理を実現できます
  4. ツールはべき等性・エラーハンドリング・適切な粒度の3原則で設計します。不可逆な操作（本番DB削除・メール送信等）はエージェントに直接与えず、Human-in-the-loopを挟む設計が必須です
  5. ループ暴走・コスト爆発・機密情報漏洩のリスクに対し、最大反復数・トークン予算・出力フィルタリングという3つの制御機構を設計段階から組み込んでください
