RAG（検索拡張生成）設計・実装ガイド

社内ドキュメントとLLMを繋ぐアーキテクチャ

16 min read 2026-03-25 RAG ベクトルDB Embedding 検索

「GPT-4に社内マニュアルの内容を聞いたら、全然違うことを答えた」——LLMを業務に組み込もうとした開発者なら一度は経験する壁です。この問題の根本は、汎用LLMがあなたの組織固有の知識を持っていないことにあります。RAG（Retrieval-Augmented Generation）は、LLMの強力な言語理解・生成能力を保ちながら、外部の最新情報・組織固有の情報を動的に注入できるアーキテクチャです。本記事では、RAGの設計から実装、品質評価まで体系的に解説します。

1. RAGが解決する問題

LLMには構造的に避けられない2つの限界があります。これを理解することが、RAGを適切に設計する第一歩です。

LLMの2つの本質的限界

第一の限界は知識カットオフ（Knowledge Cutoff）です。LLMは特定の時点までのデータで学習されており、それ以降の情報——昨日リリースされた製品仕様、先月改訂された社内規約、今朝起きた障害事例——を原理的に知ることができません。

第二の限界は幻覚（Hallucination）です。LLMは「それらしい文章」を生成するよう最適化されているため、知らないことを聞かれると、もっともらしいが事実と異なる情報を自信を持って返すことがあります。特に固有名詞・数値・手順書のような具体的事実で起きやすく、業務利用では致命的なリスクになります。

RAGはこの2つの問題を、回答生成時に関連ドキュメントを検索してコンテキストとしてLLMに渡すという仕組みで解決します。LLMは「自分の記憶」ではなく「提供された文書」をもとに回答するため、最新情報への対応と幻覚の抑制が同時に達成できます。

RAG vs ファインチューニング：選択基準

「どうせなら自社データでファインチューニングすれば良いのでは？」という疑問は自然です。しかし両者は目的が異なり、正しく使い分けることが重要です。以下のフローチャートを参考にしてください。

ファインチューニングはモデルの「振る舞い・文体・推論スタイル」を変えることに適しています。一方、RAGは「特定の事実・最新情報・組織固有の知識」を提供するのに適しており、データが頻繁に更新される場面では圧倒的にコスト効率が良い選択です。

RAGが特に有効なシナリオ

  • 社内ナレッジベース — 過去の障害対応事例、設計書、議事録などを参照した質問応答
  • 法令・社内規約の参照 — 頻繁に改訂される規約文書をリアルタイムに参照する必要がある場面
  • 製品マニュアル・FAQ — カスタマーサポート自動化で正確な製品情報を返す必要がある場合
  • コードリポジトリ検索 — 社内ライブラリの使い方やアーキテクチャ決定記録（ADR）の照会

2. RAGアーキテクチャ全体図

RAGは大きく2つのパイプラインで構成されます。ドキュメントを事前にベクトル化してDBに格納するインデキシングパイプラインと、ユーザーの質問に答えるために関連文書を検索してLLMに渡すクエリパイプラインです。

コンポーネント一覧

コンポーネント 役割 主要ツール例
Document Loader 文書読み込み・テキスト抽出 LangChain, LlamaIndex
Text Splitter 文書をチャンクに分割 RecursiveCharacterTextSplitter
Embedding Model テキスト→ベクトル変換 text-embedding-3-small, Cohere
Vector Store ベクトル格納・類似検索 pgvector, Chroma, Pinecone
LLM コンテキストをもとに回答生成 GPT-4o, Claude, Gemini

3. Embeddingとベクトル検索

Embeddingの直感的理解

Embeddingとは、テキストを数百〜数千次元の浮動小数点ベクトルに変換する技術です。直感的なイメージとして、意味が近い文書ほどベクトル空間上での距離が近くなるように設計されています。「犬」と「イヌ」は非常に近い位置に、「犬」と「飛行機」は遠い位置にマッピングされます。

この性質を利用することで、「キーワードの完全一致」ではなく「意味の類似度」による検索が可能になります。ユーザーが「接続が切れる」と質問しても、「タイムアウトエラーの対処法」というチャンクを正しく取得できるのはこのためです。

コサイン類似度

ベクトル間の類似度の測定には、主にコサイン類似度が使用されます。値の範囲は −1〜1 で、1に近いほど意味が類似しており、0は無相関、−1は正反対の意味を持ちます。ほとんどの実用シナリオでは 0.7〜0.9 の範囲が「十分に関連している」と判断できる閾値になります。この閾値はドメインとデータによって調整が必要です。

Vector DB 比較

DB タイプ 特徴 推奨シナリオ
pgvector PostgreSQL拡張 既存PG環境と統合、SQL で操作可能 中小規模・既存PostgreSQL環境
Chroma OSS組込み型 軽量・Python native・インストール簡単 開発・PoC・ローカル検証
Pinecone クラウドマネージド スケーラブル・インフラ管理不要 大規模・マネージドサービス希望
Azure AI Search クラウドマネージド Azure統合・ハイブリッド検索対応 Azureベースの開発環境
Amazon OpenSearch クラウドマネージド AWS統合・フルテキスト検索との併用 AWSベースの開発環境

Embeddingの生成とベクトルDBへの格納

python — Embeddingの生成とpgvectorへの格納 
from openai import OpenAI
import psycopg2
import json

client = OpenAI()

def embed_text(text: str) -> list[float]:
    """テキストをEmbeddingベクトルに変換する"""
    response = client.embeddings.create(
        model="text-embedding-3-small",
        input=text
    )
    return response.data[0].embedding

def store_chunk(conn, chunk_id: str, content: str, metadata: dict) -> None:
    """チャンクとそのEmbeddingをpgvectorに格納する"""
    vector = embed_text(content)

    with conn.cursor() as cur:
        cur.execute(
            """
            INSERT INTO document_chunks (id, content, metadata, embedding)
            VALUES (%s, %s, %s, %s)
            ON CONFLICT (id) DO UPDATE
              SET content = EXCLUDED.content,
                  metadata = EXCLUDED.metadata,
                  embedding = EXCLUDED.embedding
            """,
            (chunk_id, content, json.dumps(metadata), vector)
        )
    conn.commit()

def search_similar(conn, query: str, top_k: int = 5) -> list[dict]:
    """クエリと意味的に近いチャンクを取得する"""
    query_vector = embed_text(query)

    with conn.cursor() as cur:
        cur.execute(
            """
            SELECT id, content, metadata,
                   1 - (embedding <=> %s::vector) AS similarity
            FROM document_chunks
            ORDER BY embedding <=> %s::vector
            LIMIT %s
            """,
            (query_vector, query_vector, top_k)
        )
        rows = cur.fetchall()

    return [
        {"id": r[0], "content": r[1], "metadata": r[2], "similarity": r[3]}
        for r in rows
    ]

ハイブリッド検索について

ベクトル検索（意味類似）とBM25キーワード検索（完全一致・部分一致）を組み合わせたハイブリッド検索は、実務上の精度向上に効果的です。例えば製品型番「XR-2400-JP」のような固有識別子はベクトル検索で拾いにくく、キーワード検索が有効です。一方、「接続が不安定」のような自然言語の質問はベクトル検索が強みを発揮します。Azure AI SearchやAmazon OpenSearchはハイブリッド検索をネイティブサポートしており、RRF（Reciprocal Rank Fusion）などで両方のスコアをマージできます。

4. チャンキング戦略

RAGの精度を左右する最重要要素の一つがチャンキング（文書の分割方法）です。分割が粗すぎるとLLMのコンテキストウィンドウを圧迫し、細かすぎると必要な文脈が欠落して回答精度が下がります。どちらも「良さそうな文書は見つかったのに回答がおかしい」という症状として現れます。

チャンキング戦略の比較

戦略 方法 メリット デメリット
固定長 500文字ごとに機械的に分割 実装がシンプル・高速 文・段落の途中で切れる場合がある
文境界 句点（。）や改行で分割 意味の単位を保持しやすい チャンクサイズが不均一になる
セマンティック Embeddingの類似度変化点で分割 意味のまとまりを高精度で保持 計算コストが高く実装が複雑
Hierarchical セクション→段落→文の階層構造 文書の論理構造を保持できる 実装と管理が複雑、DB設計が必要

日本語文書特有の課題

日本語ドキュメントをRAGに組み込む際は、英語ドキュメントにはない以下の課題があります。

  • PDFからの抽出品質 — 日本語PDFはフォント埋め込みや文字コードの問題で文字化けやレイアウト崩れが頻発します。pdfplumberやpymupdfの使い分け、またはAzure Document Intelligenceのような高精度OCRサービスの活用を検討してください
  • 縦書き・複雑レイアウト — 縦書きや2段組のPDFは一般的なテキスト抽出では読み順が崩れます。書類のフォーマットに合った抽出ツールの選定が必要です
  • ルビ・記号 — ルビ（読み仮名）が別テキストとして抽出されたり、全角・半角の混在が検索精度に影響します。前処理での正規化が重要です
  • 文節・形態素境界 — 日本語は単語間にスペースがないため、固定長での分割が特に意味単位を壊しやすいです。mecabやSudachiを使った形態素解析での分割も選択肢になります

推奨設定値

python — LangChain RecursiveCharacterTextSplitterの推奨設定 
from langchain_text_splitters import RecursiveCharacterTextSplitter

# 日本語文書向け推奨設定
# chunk_size: 検索精度とコンテキストのバランスを考慮した400〜600文字
# chunk_overlap: 前後チャンクの文脈を繋ぐため chunk_size の10〜20%
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=80,
    separators=[
        "\n\n",   # 段落区切り（最優先）
        "\n",     # 行区切り
        "。",     # 句点
        "、",     # 読点
        " ",      # スペース
        "",       # 文字単位（フォールバック）
    ],
    length_function=len,  # 文字数カウント（日本語対応）
)

# 使用例
with open("manual.txt", encoding="utf-8") as f:
    raw_text = f.read()

chunks = splitter.split_text(raw_text)

print(f"総チャンク数: {len(chunks)}")
print(f"平均チャンクサイズ: {sum(len(c) for c in chunks) / len(chunks):.0f}文字")
print(f"最大チャンクサイズ: {max(len(c) for c in chunks)}文字")
print(f"最小チャンクサイズ: {min(len(c) for c in chunks)}文字")

chunk_size の最適値はドキュメントの性質に依存します。技術仕様書や法令文書では段落単位の 300〜500 文字が精度が高い傾向にあります。Q&Aドキュメントは1つの質問と回答のペアを1チャンクとして扱うとより高精度になります。オーバーラップは隣接チャンクの文脈を保持するために重要で、特に文末・文頭が重要な日本語では効果的です。

5. RAG品質評価

RAGパイプラインを構築して「動いた」だけでは不十分です。実運用前には定量的な品質評価が必要であり、RAGAS（RAG Assessment）はそのための標準的なフレームワークです。

RAGASの4指標

RAGASは以下の4つの観点でRAGパイプラインを評価します。

指標 定義 測定対象
Faithfulness（忠実性） 回答が取得したコンテキストに基づいているか 幻覚の有無（0〜1）
Answer Relevancy（回答関連性） 回答がユーザーの質問に対して適切か 回答の的確さ（0〜1）
Context Precision（文脈精度） 取得したコンテキストのうち関連情報の割合 検索結果のノイズ（0〜1）
Context Recall（文脈再現率） 回答に必要な情報が取得できているか 検索の網羅性（0〜1）

実用的な品質基準として、Faithfulness ≥ 0.85、Answer Relevancy ≥ 0.80 を目安にするチームが多いです。Context PrecisionとRecallはトレードオフの関係にあり、top_k（取得チャンク数）の調整でバランスをとります。

RAGASによる評価実装

python — RAGASを使ったパイプライン評価 
from datasets import Dataset
from ragas import evaluate
from ragas.metrics import (
    faithfulness,
    answer_relevancy,
    context_precision,
    context_recall,
)

# 評価用データセット
# 各サンプルは question, answer, contexts, ground_truth を持つ
eval_data = {
    "question": [
        "有給休暇の申請期限はいつですか？",
        "テレワーク勤務の上限日数は？",
    ],
    "answer": [
        # RAGパイプラインが生成した回答（実際には pipeline.run(q) で取得）
        "有給休暇は取得日の3営業日前までに申請が必要です。",
        "テレワーク勤務は月15日を上限としています。",
    ],
    "contexts": [
        # 検索で取得されたチャンクリスト
        ["就業規則第12条：有給休暇の申請は取得希望日の3営業日前までに...", ],
        ["テレワーク勤務規程第5条：在宅勤務の上限は月15日とする...", ],
    ],
    "ground_truth": [
        # 期待される正解（人手で用意するゴールドスタンダード）
        "有給休暇は取得日の3営業日前までに申請が必要です。",
        "テレワーク勤務は月15日を上限としています。",
    ],
}

dataset = Dataset.from_dict(eval_data)

# 評価実行
results = evaluate(
    dataset=dataset,
    metrics=[
        faithfulness,
        answer_relevancy,
        context_precision,
        context_recall,
    ],
    llm=...,          # 評価用LLM (OpenAI GPT-4o 等)
    embeddings=...,   # 評価用Embedding Model
)

print(results)
# 出力例:
# {'faithfulness': 0.92, 'answer_relevancy': 0.87,
#  'context_precision': 0.78, 'context_recall': 0.83}

納品前のQAチェックリスト

  • [ ] 幻覚率（Faithfulness）が 0.85 以上であることを確認
  • [ ] 検索精度（Context Precision）が 0.75 以上であることを確認
  • [ ] 回答網羅性（Context Recall）が 0.80 以上であることを確認
  • [ ] テストセットに「DBに存在しない質問」（アウトオブドメイン）を含め、適切に「情報がありません」と返すか検証
  • [ ] 個人情報・機密情報を含む文書が意図せず返答に漏洩しないか確認
  • [ ] レイテンシが実用範囲内（P95 < 3秒程度）であることを測定
  • [ ] チャンクが正しく分割されているか（意味の途中で切れていないか）を人手でサンプリング確認

継続的改善のサイクル

RAGの品質は初期構築で完結するものではありません。本番運用後は以下のサイクルで継続的に改善します。

  1. ログ収集 — 質問・取得チャンク・回答・ユーザーフィードバックを記録する
  2. 低スコア事例の分析 — Faithfulnessが低い回答や「情報が見つかりません」の頻度を分析し、原因（チャンキング不良、Embedding精度、クエリ処理）を特定する
  3. 改善策の実施 — チャンクサイズ調整、クエリ拡張（LLMでクエリを言い換えて複数回検索）、リランカーの追加など
  4. 評価セットでの検証 — 変更後にRAGASで回帰テストを実施し、スコアが改善していることを確認
  5. 本番反映 & 監視 — A/Bテストで段階的にロールアウトし、本番スコアをモニタリングする

この記事のまとめ

  1. RAGはLLMの知識カットオフと幻覚という2つの構造的限界を、回答時に関連文書を動的に注入することで解決します。知識の更新頻度が高い場合はファインチューニングよりRAGが適切です
  2. RAGはインデキシングパイプライン（文書→チャンキング→Embedding→ベクトルDB）とクエリパイプライン（質問→類似検索→コンテキスト挿入→LLM）の2構成です
  3. Vector DBの選定はスケール・既存インフラ・マネージド要件で決める。中小規模の既存PostgreSQL環境にはpgvector、開発・PoC段階にはChromaが最初の選択肢として実用的です
  4. チャンキングはRAG品質の最重要要素の一つ。日本語文書では RecursiveCharacterTextSplitter の separators に句点・読点を含め、chunk_size 500文字・overlap 80文字が現実的な出発点です
  5. 納品前にRAGASでFaithfulness・Answer Relevancy・Context Precision・Context Recallの4指標を定量評価し、継続的な改善サイクルを構築することが本番運用の鍵です
