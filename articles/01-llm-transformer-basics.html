LLM・トランスフォーマーの基礎理論

生成AIはなぜ「賢く」見えるのか — 仕組みから理解するLLM

14 min read 2026-03-25 LLM Transformer 基礎理論 アーキテクチャ

1. 言語モデルとは何か

「ChatGPTは本当に文章を理解しているのか？」——この問いに対する正直な答えは「理解という概念次第」です。LLM（大規模言語モデル）の本質を一言で表すなら、確率的な次トークン予測マシンです。過去のトークン列を入力として受け取り、次に来る可能性が最も高いトークンを出力する。その繰り返しによって文章が生成されます。

この単純に聞こえる仕組みが、なぜ高品質なコード生成や複雑な推論を実現できるのか——それを理解するには、まずトークンとモデルの構造を把握する必要があります。

トークンとは何か

LLMはテキストを文字や単語の単位ではなく、トークンという独自の断片として処理します。トークナイザーの実装によって異なりますが、GPT-4のBPE（Byte Pair Encoding）では英語の場合おおよそ1単語≈0.75トークン、日本語では漢字1文字が1〜2トークンになることが多いです。

  • 単語ベース: "hello" → hello（1トークン）
  • サブワードベース（BPE）: "tokenization" → token + ization（2トークン）
  • 文字ベース: "AI" → A + I（2トークン）

なぜトークン単位にするかというと、語彙サイズ（ボキャブラリー）と表現力のトレードオフを最適化するためです。文字単位では語彙が小さくなりモデルが軽くなる一方、同じ意味を表すのに多くのステップが必要になります。単語単位では未知語（out-of-vocabulary）問題が深刻化します。サブワード方式はその中間点として実用上最も広く採用されています。

数式的な直感：条件付き確率の連鎖

LLMがやっていることを数式で表すと以下のようになります。

math 
P(t₁, t₂, ..., tₙ) = P(t₁) × P(t₂|t₁) × P(t₃|t₁,t₂) × ... × P(tₙ|t₁,...,tₙ₋₁)

// 推論時：最も確率の高い次のトークンを選ぶ（greedy decoding）
t̂ₙ = argmax P(tₙ | t₁, t₂, ..., tₙ₋₁)

あるトークン列が与えられたとき、次のトークンの確率分布を出力する——これが全てです。「なぜGPTが賢く見えるか」は、巨大なデータと計算資源で訓練されたこの確率分布の精度が、人間の書いた文章と非常に近くなっているからです。

たとえば「東京の天気は」というトークン列が入力された場合、モデルは次のトークンとして「晴れ」「雨」「曇り」「今日」などに高い確率を割り当て、「バナナ」や「量子」には低い確率を割り当てます。これは単純なパターンマッチングではなく、文脈全体を考慮した確率推定です。

2. トランスフォーマーアーキテクチャ

2017年のGoogle論文「Attention Is All You Need」で提案されたトランスフォーマーは、それ以前のRNN（再帰型ニューラルネットワーク）が抱えていた長距離依存関係の学習困難という問題を根本的に解決しました。現代のLLMは事実上すべてこのアーキテクチャをベースにしています。

Attention機構の直感的理解

「猫が魚を食べた」という文を処理するとき、「食べた」という動詞は「猫」（主語）と「魚」（目的語）の両方と強く関連しています。Attention（注意）機構は、各トークンが他のどのトークンに「注目」すべきかを重み付けで学習する仕組みです。

この重みはQuery（クエリ）・Key（キー）・Value（バリュー）という3つのベクトルを使って計算されます。QueryとKeyの内積から注意スコアを算出し、Softmaxで正規化したものをValueに掛け合わせることで、文脈を考慮した表現が得られます。

3種類のアーキテクチャ

トランスフォーマーは用途に応じて3つの構成に分類されます。

  • Encoder-only（例: BERT, RoBERTa）
    入力テキスト全体を双方向に処理し、各トークンの文脈表現を得ます。テキスト分類・固有表現認識・文書検索（埋め込み生成）に優れますが、テキスト生成には向きません。
  • Decoder-only（例: GPT-4, Claude, Llama）
    過去のトークンのみを参照する自己回帰的な生成を行います。現在の主流です。プロンプトに対して逐次的にトークンを生成します。
  • Encoder-Decoder（例: T5, BART）
    Encoderが入力を圧縮表現に変換し、DecoderがCross-Attentionを使ってその表現から出力を生成します。機械翻訳・要約など入出力の長さが異なるタスクに適します。

なぜDecoder-onlyが主流になったか

Decoder-onlyモデルが席巻した最大の理由はスケーラビリティです。事前学習タスク（次トークン予測）とファインチューニングのタスク（指示への応答）が同一のフォーマット上で動作するため、同じアーキテクチャのまま規模を拡大するだけで多様な能力が emergent（創発）します。Encoder-Decoderは2つのコンポーネントを別々に管理する必要があり、スケールアップが複雑になります。

マルチヘッドアテンション（Multi-Head Attention）

実際のトランスフォーマーでは、Attentionを1つではなく複数の「ヘッド」で並列実行します。各ヘッドは異なる「注目パターン」を学習できます——あるヘッドは構文的関係（主語と動詞）に注目し、別のヘッドは意味的関係（代名詞と先行詞）に注目するといった具合です。ヘッド数はモデルによって異なり（GPT-3では96層・96ヘッド）、これらの多様な視点が組み合わさることで豊かな言語理解が実現します。なお、GPT-4の詳細なアーキテクチャはOpenAIにより非公開とされています。

3. 学習の3ステップ

ChatGPTやClaudeのようなモデルが「指示に従い、安全で有用な回答を返す」ようになるまでには、大きく3つの学習フェーズがあります。それぞれのフェーズがモデルの性質を根本的に変えていきます。

Step 1: 事前学習（Pre-training）

インターネット上のWebページ、書籍、コード、論文など、人類が生成した膨大なテキストデータを使い、自己教師あり学習でモデルを訓練します。「正解ラベル」は不要で、入力テキストの次のトークンがそのまま正解になります。GPT-4の事前学習には数兆トークン・数千億円規模の計算コストがかかったとされます。

この段階のモデルは「テキスト補完機」です。「東京は」と入力すれば「日本の首都で」と続けますが、「要約してください」という指示には適切に応答できません。

Step 2: 教師ありファインチューニング（Supervised Fine-tuning / SFT）

人間が作成した（指示, 理想的な応答）のペアを使い、モデルを指示に従えるようチューニングします。「〜してください」という指示に対して「はい、〜します」と応答する振る舞いを学ばせます。データ量は事前学習に比べて桁違いに少なく（数万〜数十万件程度）、コストも低い一方で効果は劇的です。

Step 3: RLHF（人間のフィードバックによる強化学習）

SFTまでのモデルでも指示には従えますが、有害な内容を生成しない・ユーザーに本当に有用な回答を返すという点は不完全です。RLHFでは以下の2フェーズを踏みます：

  1. 報酬モデルの学習: 同じ質問に対する複数の回答を人間がランク付けし、その好みを学習した「報酬モデル（Reward Model）」を構築する
  2. PPOによる強化学習: 報酬モデルのスコアを最大化するようにLLMをさらに更新する。ただしオリジナルモデルから大きく逸脱しないようKLダイバージェンスで制約をかける

最近ではRLHFの代替としてDPO（Direct Preference Optimization）が注目されており、報酬モデルを別途構築せず直接選好データからファインチューニングできるため、実装がシンプルになります。

4. スケーリング則とモデル比較

スケーリング則（Scaling Laws）

2020年にOpenAIが発表した研究により、LLMの性能（損失）はモデルのパラメータ数・学習データ量・計算量（FLOPs）のべき乗則に従って改善することが示されました。これをスケーリング則（Scaling Laws）と呼びます。

重要な含意は、「より大きいモデル・より多いデータ・より多い計算」が予測可能な形で性能を向上させるという点です。ただし2022年のChinchillaの研究では、同じ計算予算ならモデルサイズと学習データ量を均衡させるべきことが示され、GPT-3系は相対的にデータ量が少なすぎた可能性が指摘されました。

コンテキストウィンドウ

モデルが一度に「見られる」トークン数をコンテキストウィンドウ（Context Window）と呼びます。この上限を超えた過去のトークンはモデルから見えなくなります。2023年以前は8K〜32K tokens程度が上限でしたが、2026年現在では100K〜1M tokensが一般的になりつつあります。

コンテキストが長いと、長大な仕様書をまるごと与えてのコード生成や、会話履歴全体を保持したままの対話が可能になります。一方でコンテキスト長が増えるほど推論コスト（計算量・料金）も増加することに注意が必要です。

主要モデル比較（2025年末時点）

モデル 提供元 コンテキスト 特徴
gpt-4o OpenAI 128K tokens 高精度、マルチモーダル（画像・音声対応）、広いエコシステム
claude-3-7-sonnet Anthropic 200K tokens 長文理解・コーディング特化、思考モード（拡張推論）対応
gemini-2.0-flash Google 1M tokens 超高速・低コスト、Google Workspaceとの連携、長文処理に強み
llama-3.3-70b Meta 128K tokens オープンソース、オンプレミス・プライベートクラウド運用可、商用利用可

どのモデルを選ぶかは「精度・速度・コスト・データ管理ポリシー」の4要素のバランスで決まります。特に機密性の高い業務では、データが外部サーバーに送信されないオンプレミス運用（Llama等のオープンウェイトモデル）が選ばれるケースが増えています。

5. 実務への接続：APIとコスト

LLMの理論を把握したら、次は実際にAPIとして組み込む際の基礎を押さえましょう。ほとんどのLLMプロバイダーはRESTful APIを提供しており、HTTPリクエストでモデルを呼び出せます。

基本的なAPI呼び出し構造

bash 
# 汎用的なLLM API呼び出しの例（Chat Completions形式）
curl https://api.example-llm.com/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $API_KEY" \
  -d '{
    "model": "your-model-name",
    "messages": [
      {
        "role": "system",
        "content": "あなたは経験豊富なソフトウェアアーキテクトです。"
      },
      {
        "role": "user",
        "content": "マイクロサービスとモノリスの選択基準を教えてください。"
      }
    ],
    "temperature": 0.3,
    "max_tokens": 1024
  }'

レスポンスは choices[0].message.content にテキストが入り、usage フィールドに消費トークン数が記録されます。このトークン数が課金の基準になります。

コスト計算の基礎

API利用料は 入力トークン数 × 入力単価 + 出力トークン数 × 出力単価 で決まります。一般的に出力トークンの単価は入力の2〜4倍に設定されています（生成処理は推論負荷が高いため）。

python 
# コスト試算の例（料金は変動するため必ずプロバイダーの最新情報を確認）
def estimate_cost(
    input_tokens: int,
    output_tokens: int,
    input_price_per_1m: float,   # 例: 3.0 USD/1M tokens
    output_price_per_1m: float,  # 例: 15.0 USD/1M tokens
) -> float:
    input_cost  = (input_tokens  / 1_000_000) * input_price_per_1m
    output_cost = (output_tokens / 1_000_000) * output_price_per_1m
    return input_cost + output_cost

# 例: 1000トークン入力・500トークン出力を1万回/日実行した場合
daily_cost = estimate_cost(
    input_tokens=1_000 * 10_000,
    output_tokens=500 * 10_000,
    input_price_per_1m=3.0,
    output_price_per_1m=15.0,
)
print(f"日次コスト: ${daily_cost:.2f}")  # → 約$105/日

レートリミットとレイテンシの考慮事項

本番環境でAPIを組み込む際に見落としがちな非機能要件があります。

  • RPM / TPM（Requests/Tokens per Minute）: プランによって上限が決まっており、バースト的なトラフィックでは429エラーが返る。Exponential backoffによるリトライ実装は必須です。
  • TTFT（Time to First Token）: 最初のトークンが返るまでの時間。ユーザー向けUIではストリーミング出力（Server-Sent Events）を使い、体感レイテンシを下げる設計が重要です。
  • TPS（Tokens per Second）: 生成速度。長い応答を生成する場合の全体レスポンスタイムを試算しておきましょう。

本番利用前に確認すべき3つのこと

  1. データ保存・学習利用ポリシー: 送信したプロンプトがモデルの再学習に使われるか。API利用規約のデータ利用条項を確認し、必要に応じてオプトアウトまたはプライベートデプロイを選択する
  2. レートリミット上限とスケールアップ申請: デフォルト上限はスタータープランで低く設定されていることが多い。本番負荷試験前にプロバイダーへの上限引き上げ申請が必要な場合がある
  3. SLA（可用性保証）: 多くのAPIは99.9%のSLAを提供するが、メンテナンスウィンドウや障害時のフォールバック戦略を設計しておく。特に複数モデルへのフォールバックは障害耐性向上に効果的

この記事のまとめ

  1. LLMは確率的な次トークン予測モデルであり、巨大なデータで訓練された条件付き確率分布が「賢さ」の正体です
  2. トランスフォーマーのAttention機構は、各トークンが文脈中の他のトークンへの「注目度」を学習することで長距離依存関係を捉えます
  3. 現在主流のDecoder-onlyアーキテクチャは、事前学習→SFT→RLHFの3段階を経て「指示に従い安全に応答する」モデルになります
  4. スケーリング則により性能はパラメータ数・データ量・計算量のべき乗で改善しますが、Chinchillaの知見からデータとモデルサイズのバランスが重要です
  5. API組み込み時はトークン単位のコスト計算・レートリミット対策・データ保存ポリシーの確認を本番前に必ず行いましょう
