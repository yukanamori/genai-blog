プロンプトエンジニアリング実践ガイド

AIへの「正しい指示の出し方」は設計スキルである

12 min read 2026-03-25 プロンプト 実践 Chain-of-Thought Few-shot

「ChatGPTに聞いたけど使えない回答しか返ってこない」——この経験をしたことがある開発者は多いはずです。その原因の大半は、モデルの限界ではなくプロンプトの設計にあります。プロンプトエンジニアリングとは単なる「魔法の言葉を覚える」スキルではなく、モデルに与える入力仕様を設計する能力です。要件定義や API 設計と同じように、精度・再現性・保守性を意識した設計が求められます。

1. プロンプトの構造分解

Chat Completions 形式の API では、1 回の呼び出しは複数のメッセージで構成されます。それぞれが異なる役割を担い、この構造を理解することがプロンプト設計の出発点です。

各レイヤーの使い分け

System プロンプトはモデルの振る舞いを規定する「設定ファイル」です。ここに書くべき内容は以下の通りです：

  • ペルソナ定義（「あなたはXXの専門家です」）
  • 出力フォーマット指定（「必ずJSON形式で返してください」）
  • 制約条件（「コード以外の説明文は含めないでください」）
  • Few-shot 例のコンテキスト（後述）

User メッセージには実際のタスクを記述します。長い仕様書やコードをここに貼り付けるのが一般的です。Assistant ターンは多ターン会話で過去の応答を履歴として渡すか、Few-shot の「良い応答例」をあらかじめ埋め込む目的で使います。

2. 主要テクニック

Zero-shot：例なし直接指示

最もシンプルな形式です。モデルに例を与えず、タスクを直接記述します。GPT-4 / Claude 3.7 のような高性能モデルは Zero-shot でも多くのタスクをこなせますが、出力フォーマットの指定が曖昧なままだと構造がブレます。

json 
// Zero-shot: フォーマットを明示して指示する
{
  "role": "user",
  "content": "次の要件をUMLのユースケース図に変換してください。\nPlantUML形式で出力し、コードブロックのみを返してください。\n\n要件: ユーザーが商品を検索し、カートに追加し、購入できる"
}

Few-shot：例を2〜5個与える

モデルに「正解のパターン」を見せることで、出力の一貫性と精度が大幅に向上します。特に独自の出力フォーマット・用語定義・業務固有の分類基準がある場合に有効です。例は多ければ良いわけではなく、2〜5件が実用上の最適解です（例が多すぎるとコンテキストを圧迫し、かえってモデルが混乱することがある）。

json 
// Few-shot: バグレポートをJSON構造化するエージェント
{
  "role": "system",
  "content": "入力されたバグレポートを構造化JSONに変換してください。\n必ず以下のキーを含めてください: component, severity, steps\n\n例1:\n入力: ログイン画面でパスワードを入力すると500エラーが出る\n出力: {\"component\": \"認証\", \"severity\": \"Critical\", \"steps\": [\"ログイン画面を開く\", \"パスワードを入力する\", \"ログインボタンを押下する\"]}\n\n例2:\n入力: 検索結果が0件のとき画面が真っ白になる\n出力: {\"component\": \"検索\", \"severity\": \"Major\", \"steps\": [\"検索画面を開く\", \"ヒットしないキーワードを入力する\", \"検索ボタンを押下する\"]}"
},
{
  "role": "user",
  "content": "注文確認メールが届かないという問い合わせが複数件来ています"
}

Chain-of-Thought（CoT）：推論ステップを明示させる

「ステップバイステップで考えてください」という一文を加えるだけで、複雑な推論タスクの精度が大幅に向上します。これは CoT（Chain-of-Thought）プロンプティングと呼ばれ、2022年のGoogle Brain の論文で実証された手法です。

なぜ効くのか：LLM はトークンを逐次生成するため、中間的な思考過程をテキストとして出力させることで、モデル自身が「考えながら答える」状態を作り出せます。特に計算・論理推論・複数条件の組み合わせが必要なタスクで威力を発揮します。

json 
// Chain-of-Thought: 設計判断の根拠を段階的に出力させる
{
  "role": "system",
  "content": "あなたは経験豊富なソフトウェアアーキテクトです。\n設計判断を行う際は必ず以下の順序で考えてください:\n1. 要件の整理と制約条件の列挙\n2. 候補となるアーキテクチャパターンの列挙\n3. 各パターンのトレードオフ分析\n4. 推奨案とその理由"
},
{
  "role": "user",
  "content": "日次バッチ処理（件数: 約50万件/日）をリアルタイムAPI処理に移行する設計を検討してください。\n現行: PostgreSQL直接クエリのバッチ\n要件: レイテンシ < 200ms、可用性99.9%以上"
}

3. 業務別プロンプトパターン

開発プロセスの各フェーズにおいて、LLMが最も価値を発揮できるユースケースとプロンプトパターンは異なります。以下の表を参考に、フェーズごとに適切なアプローチを選択してください。

業務フェーズ プロンプトパターン 効果・活用例
要件定義 ユーザーストーリー抽出 ヒアリング議事録・RFP文書 → 構造化された要件一覧・受け入れ条件の生成
基本設計 ER図・シーケンス設計支援 要件テキスト → PlantUML/Mermaidのドラフト、設計書のレビュー指摘
実装 コード生成・補完 API仕様・型定義 → ボイラープレートコード、リファクタリング提案
テスト テスト仕様・ケース生成 設計書・仕様書 → 境界値・異常系を含むテストケース一覧
障害対応 根本原因分析（RCA） スタックトレース・ログ → 原因候補の列挙と調査手順の優先度付け

要件定義フェーズのプロンプトテンプレート

以下は要件定義フェーズで実際に使えるプロンプトテンプレートです。ヒアリング議事録や RFP テキストをそのまま貼り付けて活用できます。

json 
{
  "role": "system",
  "content": "あなたは要件定義の専門家です。以下の手順で入力テキストを処理してください。\n\n【処理手順】\n1. 利害関係者（ステークホルダー）を列挙する\n2. 各ステークホルダーの目的・ゴールを抽出する\n3. ユーザーストーリー形式（「〜として、〜したい。なぜなら〜だから」）で要件を記述する\n4. 各要件に優先度（Must/Should/Could/Won't）を付与する\n5. 曖昧・不明確な箇所を【確認事項】として列挙する\n\n【出力形式】\n- Markdown形式\n- ユーザーストーリーには連番IDを付与（US-001, US-002...）\n- 確認事項は番号付きリストで末尾に記載"
},
{
  "role": "user",
  "content": "以下のヒアリング議事録から要件を整理してください。\n\n---\n[議事録テキストをここに貼り付け]\n---"
}

障害対応（RCA）プロンプトテンプレート

障害対応では時間的プレッシャーが高いため、構造化された分析を素早く出力させるプロンプトが有効です。

json 
{
  "role": "system",
  "content": "あなたはインシデント対応の専門家です。提供されたエラーログ・スタックトレースを分析し、以下のフォーマットで回答してください。\n\n## 原因候補（確率順）\n1. [最有力候補] 説明と根拠\n2. [次点] 説明と根拠\n\n## 即時調査手順\n- 確認すべきログ・メトリクス（具体的なコマンドを含む）\n\n## 暫定対処\n- サービス影響を最小化するための即時対応案\n\n## 恒久対処候補\n- 根本原因が特定された後の修正方針\n\n推測が含まれる場合は「（推測）」と明記してください。"
},
{
  "role": "user",
  "content": "以下のスタックトレースとアクセスログを分析してください。\n\n[ERROR] 2026-03-25 14:32:01 - java.lang.OutOfMemoryError: GC overhead limit exceeded\n  at java.util.Arrays.copyOf(Arrays.java:3210)\n  at com.example.service.ReportService.generateCSV(ReportService.java:234)\n  at com.example.controller.ReportController.export(ReportController.java:89)\n\nアクセスログ: 14:31に/api/report/exportへの同時リクエスト数が通常の8倍に急増"
}

4. 失敗パターンと修正

現場でよく見る3大失敗パターン

  1. 曖昧な指示: 「いい感じに書いて」「わかりやすくして」など、評価基準が主観的。モデルは「いい感じ」の定義を推測するしかなく、期待と外れた出力になりやすい
  2. 矛盾する制約: 「短く簡潔に、かつ詳細に」「初心者向けに、ただし専門用語も使って」などのトレードオフを解決せずに両方を要求する
  3. コンテキスト不足: 「このコードを修正して」と言うだけで、言語・フレームワーク・修正の目的・制約条件を伝えない

失敗例 vs 改善例：コード生成

json — BAD: 曖昧・コンテキスト不足 
// NG: 何を直せばいいかモデルが推測できない
{
  "role": "user",
  "content": "このコードを直してください。\n\nfunction calc(a, b) {\n  return a / b;\n}"
}
json — GOOD: 目的・制約・フォーマットを明示 
// OK: 問題の具体的内容・言語バージョン・期待する変更内容を全て含む
{
  "role": "system",
  "content": "TypeScript（strict mode）のコードレビュアーです。修正後のコードのみを返し、変更箇所にはインラインコメントで理由を記載してください。"
},
{
  "role": "user",
  "content": "以下のTypeScript関数にはゼロ除算と型安全性の問題があります。\n修正してください。\n\n【問題点】\n- bが0のとき Infinity が返る（呼び出し元がエラーハンドリングを期待している）\n- 戻り値の型が number だが、エラーケースで例外をスローするか Result 型を返すべき\n\n【制約】\n- throw は使わず Result 型で返してください\n- 既存の関数シグネチャは変更しないでください\n\n```typescript\nfunction calc(a: number, b: number): number {\n  return a / b;\n}\n```"
}

失敗例 vs 改善例：矛盾する制約

json — BAD: 矛盾する指示 
// NG: 「短く」と「全部入れて」は両立しない
{
  "role": "user",
  "content": "マイクロサービスアーキテクチャの説明を短く、でも全ての概念を網羅して、わかりやすく書いてください"
}
json — GOOD: 優先順位を明示してトレードオフを解消 
// OK: 対象読者・文字数・カバーする範囲を優先順位付きで指定する
{
  "role": "system",
  "content": "技術ドキュメントライターです。指定された形式で正確に記述してください。"
},
{
  "role": "user",
  "content": "対象読者: モノリシックアーキテクチャの経験はあるが、マイクロサービスは未経験の開発者\n文字数: 400字以内（優先）\nカバーする概念: 定義・モノリスとの最大の違い・採用時の主なトレードオフの3点のみ\n\n上記の条件でマイクロサービスアーキテクチャの説明を書いてください。"
}

5. パラメータ制御とバージョン管理

Temperature：出力の「ランダム性」を制御する

Temperature は LLM が次のトークンを選ぶ際の確率分布の「広がり」を制御するパラメータです。0 に近づくほど最確率のトークンが選ばれやすくなり（決定論的）、高い値ほど低確率のトークンも選ばれやすくなります（多様・創造的）。

  • 0〜0.2: ほぼ決定論的。同じ入力に対して毎回ほぼ同じ出力。コード生成・SQL・フォーマット変換に適する
  • 0.3〜0.7: バランス型。多少のバリエーションがありつつ品質が安定。要件整理・文書要約・分析に適する
  • 0.7〜1.5: 創造的。多様なアイデアを生成。ブレインストーミング・命名案生成・マーケティングコピーに適する
  • 1.5〜2.0: 極めてランダム。実用的なシナリオでの使用は限定的。実験・創作領域（※ Anthropic Claude APIの有効範囲は0〜1.0）

Max Tokens と Stop Sequences

max_tokens は出力の上限トークン数を制限します。出力が途中で切れてもコストが発生するため、期待する出力長の1.3〜1.5倍程度に設定するのが経験則です。

stop（Stop Sequences）は指定した文字列が出力に現れた時点で生成を停止させます。Few-shot の各例の区切り文字として使うことで、余分な生成を防げます。

業務用途別 推奨パラメータ設定

業務用途 Temperature 理由
コード生成・SQL・フォーマット変換 0.1 〜 0.3 再現性・一貫性を最優先。同じ仕様から毎回同じコードが生成されることが望ましい
要件整理・議事録要約・ドキュメント生成 0.3 〜 0.5 内容の正確性を保ちつつ、表現に若干の柔軟性を持たせる
設計レビュー・リスク分析・意見出し 0.5 〜 0.7 同一観点の繰り返しを避け、複数の視点を引き出す
ブレインストーミング・命名案・アイデア出し 0.7 〜 1.0 多様性を最大化。候補を多数生成して人間が選別する前提

プロンプトを Git で管理する

本番運用では、プロンプトはコードと同等の成果物として管理すべきです。以下のベストプラクティスを推奨します。

text — 推奨ディレクトリ構造 
prompts/
├── requirements/
│   ├── user-story-extraction.v1.json    # 安定版
│   ├── user-story-extraction.v2.json    # 改善テスト中
│   └── CHANGELOG.md                     # 変更履歴
├── code-review/
│   ├── typescript-review.v1.json
│   └── typescript-review.v1.test.jsonl  # テストケース
└── incident/
    └── rca-analysis.v1.json
python — プロンプトのA/Bテスト実装例 
import json
import hashlib
from pathlib import Path

def load_prompt(name: str, version: str = "v1") -> dict:
    """バージョン指定でプロンプトを読み込む"""
    path = Path(f"prompts/{name}.{version}.json")
    return json.loads(path.read_text(encoding="utf-8"))

def ab_test_prompt(request_id: str, name: str) -> dict:
    """
    request_id のハッシュ値でA/Bを振り分ける。
    同一ユーザーには常に同じバージョンが当たるため体験が一貫する。
    """
    bucket = int(hashlib.md5(request_id.encode()).hexdigest(), 16) % 100
    version = "v2" if bucket < 20 else "v1"  # 20%をv2に流す
    return load_prompt(name, version)

# 使用例
prompt = ab_test_prompt(
    request_id="user-session-abc123",
    name="requirements/user-story-extraction"
)
# → 20%の確率でv2、80%でv1が返る

プロンプトの変更を CI/CD パイプラインに組み込み、定義したテストケース（入力・期待出力のペア）でリグレッションテストを自動実行することで、プロンプト改善が既存の動作を壊していないかを継続的に確認できます。

この記事のまとめ

  1. プロンプトはSystem / User / Assistantの3レイヤー構造を理解し、各レイヤーの役割に合った内容を配置することが基本です
  2. Zero-shot・Few-shot・CoTの3テクニックは用途で使い分ける。出力フォーマットが重要な場合はFew-shot、推論が必要な場合はCoTが効果的です
  3. 失敗の大半は曖昧な指示・矛盾する制約・コンテキスト不足の3パターン。対象読者・文字数・優先順位・制約を明示することで解消できます
  4. Temperatureはタスクの性質に合わせて設定する。コード生成は0.1〜0.3、ブレストは0.7〜1.0が目安です
  5. プロンプトはコードと同等の資産としてバージョン管理・テスト・A/Bテストを行い、継続的に改善する体制を整えましょう
